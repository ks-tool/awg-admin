# Amnezia Admin (awg-admin)

[English version](README.md)

**Amnezia Admin** — инструмент администрирования серверов [AmneziaWG](https://github.com/amnezia-vpn/amneziawg-go)/WireGuard, подключённых к ним VPN-пиров и пользователей, которым эти пиры принадлежат. Один экземпляр админки может управлять сразу несколькими серверами; на каждом сервере работает небольшой агент, который владеет реальными WireGuard-интерфейсами на этой машине.

> 📖 **Полное руководство по функционалу:** [docs/GUIDE_ru.md](docs/GUIDE_ru.md) — подробный разбор всех возможностей.

## Для чего это нужно

- Создавать и настраивать WireGuard/AmneziaWG-интерфейсы на удалённых серверах без ручного SSH — адреса, ключи и параметры обфускации AmneziaWG подставляются автоматически.
- Управлять пользователями и выдавать/отзывать VPN-пиры (ключи, allowed IPs, QR-коды и готовые клиентские конфиги) для каждого пользователя.
- Развернуть агента на сервере в один клик по SSH, держать его конфигурацию синхронизированной и **сверять** (reconcile) реальное состояние агента с базой при расхождениях.
- Связывать серверы в **многоузловые туннели**, чтобы клиенты выходили в интернет через выбранную exit-ноду.
- Следить за состоянием серверов — load average, RAM и активность трафика по каждому пиру — в едином интерфейсе.
- В любой момент делать **резервную копию** всей конфигурации в переносимый файл.
- Обращаться к агентам через автоматический SSH-туннель либо напрямую по mTLS на публичном IP.
- Запускать как локальное десктопное приложение, либо как небольшой веб-сервис на своей инфраструктуре — данные остаются под вашим контролем, а не в чужом облаке.

Как всё это работает — см. [полное руководство](docs/GUIDE_ru.md).

## Из чего состоит

| Компонент | Что это | Где |
|---|---|---|
| **Admin app** | То, с чем вы непосредственно работаете. Либо desktop-приложение на Wails, либо standalone веб-сервер — одна и та же бизнес-логика и фронтенд, две разные «обёртки». | корневой Go-модуль + `frontend/` |
| **Agent** | Небольшой HTTP-демон, который разворачивается на каждом управляемом сервере. Работает с реальными WireGuard/AmneziaWG-интерфейсами через `wgctrl-go` и отдаёт CRUD API, который вызывает admin app (по умолчанию через SSH-туннель, либо напрямую по mTLS). | `agent/` (отдельный Go-модуль) |
| **Frontend** | SPA на React 19 + TypeScript. Во время выполнения определяет, работает ли он внутри desktop-обёртки Wails или общается со standalone-сервером по HTTP, и переключает транспорт соответственно — интерфейс при этом одинаковый. | `frontend/` |

Admin app хранит все свои метаданные (серверы, SSH-учётные данные, пользователей, привязки пиров) в локальной встроенной БД (boltdb, `~/.awg-admin`) — это единственный источник истины; агент просто применяет ту конфигурацию, которую ему присылают.

## Запуск

### Desktop-приложение

Скачайте сборку для своей ОС со страницы [Releases](../../releases):

- **Windows** — `amnezia-admin-amd64-installer_<версия>.exe` (инсталлятор NSIS), либо обычный `amnezia-admin_<версия>.exe`.
- **macOS** — `amnezia-admin_<версия>.dmg` (универсальный билд: Intel + Apple Silicon). Приложение не подписано/нотаризовано Apple Developer ID — при первом запуске кликните правой кнопкой по приложению → «Открыть», чтобы обойти Gatekeeper, либо выполните `xattr -cr "/Applications/Amnezia Admin.app"`.
- **Linux** — обычный бинарник (`amnezia-admin_<версия>`); требует установленных `libgtk-3` и `libwebkit2gtk-4.1`.

Без логина и без настройки — это локальное однопользовательское приложение.

### Standalone-сервер

Если нужно открывать админку из браузера, либо запускать на headless-машине.

**Бинарник:**

```sh
./awg-admin
# по умолчанию слушает :8080
```

**Docker:**

```sh
docker run -d --name awg-admin \
  -p 8080:8080 \
  -v awg-admin-data:/data \
  ghcr.io/<owner>/<repo>:latest
```

Логин по умолчанию — **admin / admin**, сразу смените его в разделе *Settings* (или через API `PUT /auth/change-credentials`). Экран входа есть только в standalone-режиме — desktop-приложение работает локально и не открыто по сети, поэтому аутентификация ему не нужна.

Конфигурация через переменные окружения:

| Переменная | Назначение |
|---|---|
| `AWG_ADMIN_ADDR` | Адрес для прослушивания, по умолчанию `:8080` |
| `AWG_ADMIN_TLS_CERT` / `AWG_ADMIN_TLS_KEY` | Поднять HTTPS со статической парой сертификат/ключ |
| `AWG_ADMIN_AUTOCERT_DOMAINS` | Список доменов через запятую для автоматического получения сертификата от Let's Encrypt (взаимоисключимо со статическим сертификатом выше; требует доступности порта 80) |
| `AWG_ADMIN_AUTOCERT_CACHE_DIR` | Куда кэшировать автоматически выпущенный сертификат, по умолчанию `$HOME/.awg-admin-autocert` |

Данные лежат в `$HOME` — файл boltdb по пути `$HOME/.awg-admin`, кэш autocert — `$HOME/.awg-admin-autocert`, а скачанные бинарники агента (если используете пресеты деплоя с локальным кэшированием — см. «Agent» ниже) — `$HOME/.awg-admin-cache`. В Docker-образе `$HOME` — это `/data`, поэтому он оформлен как монтируемый volume.

### Agent

Разворачивается автоматически из admin app (страница сервера → «Deploy agent», при наличии SSH-доступа к машине), либо вручную:

```sh
AWG_AGENT_ADDR=127.0.0.1:8080 AWG_AGENT_DB=/var/lib/awg-agent ./awg-agent
```

| Переменная | Назначение |
|---|---|
| `AWG_AGENT_ADDR` | Адрес для прослушивания, по умолчанию `127.0.0.1:8080` (loopback — рассчитан на доступ через SSH-туннель от admin app, не выставляется наружу напрямую) |
| `AWG_AGENT_DB` | Где агент хранит свои конфигурации интерфейсов |
| `AWG_AGENT_TLS_CERT` / `AWG_AGENT_TLS_KEY` / `AWG_AGENT_TLS_CLIENT_CA` | Если заданы все три — агент слушает по mTLS вместо обычного HTTP, для прямого доступа на публичном IP без SSH-туннеля |
| `AWG_AGENT_METRICS_INTERVAL` | Как часто собирать метрики CPU/RAM/сети/пиров, по умолчанию `45s` |
| `AWG_AGENT_LOG_LEVEL` | Уровень логирования, по умолчанию `info` |

Только Linux (агент управляет реальными WireGuard-интерфейсами на хосте), amd64/arm64.

### Перенос данных между инсталляциями

Если вы использовали desktop-приложение и хотите перейти на standalone-сервер (или наоборот, либо просто переехать на другую машину), используйте `awg-migrate` — он копирует содержимое файла boltdb в переносимый JSON-дамп и обратно:

```sh
awg-migrate export -db ~/.awg-admin -out dump.json   # на старой машине
awg-migrate import -db ~/.awg-admin -in dump.json    # на новой машине
```

Оба режима развёртывания уже читают один и тот же файл/формат, так что это нужно только при переезде между машинами. *Settings → Резервная копия* экспортирует тот же дамп прямо из приложения в один клик (нативный диалог сохранения или загрузка в браузере), а восстанавливается такая копия так же — импортом через `awg-migrate import` в целевую базу. См. [раздел о бэкапе в руководстве](docs/GUIDE_ru.md#бэкап-восстановление-и-перенос).

## Сборка из исходников

Требуется Go 1.26.2+ и Node 24+.

```sh
make server      # бинарник standalone веб-сервера (build/bin/)
make run-server  # собрать фронтенд + go run, для локальной проверки
make desktop     # десктопный бинарник на Wails (нужен Wails CLI: go tool wails)
make agent       # кросс-компиляция агента под Linux (build/bin/)
make migrate     # утилита экспорта/импорта awg-migrate
```
