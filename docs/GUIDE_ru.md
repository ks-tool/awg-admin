## Установка и запуск

Amnezia-WG Admin поддерживает два режима выполнения: локальное desktop-приложение и standalone веб-сервер. Оба варианта
используют одинаковый формат хранения данных и общую бизнес-логику. Выбор режима влияет только на способ взаимодействия
с приложением.

### Desktop-приложение

Desktop-версия предназначена для локального использования и не требует дополнительной настройки после установки. Все
данные сохраняются в локальной базе пользователя, а управление агентами выполняется непосредственно из приложения.

Готовые сборки публикуются на странице [Releases](../../../releases).

Для разных операционных систем доступны следующие варианты поставки:

- **Windows** — установщик `amnezia-admin-amd64-installer_<версия>.exe` (NSIS) либо переносимый исполняемый файл
  `amnezia-admin_<версия>.exe`.
- **macOS** — образ `amnezia-admin_<версия>.dmg`, содержащий универсальную сборку для процессоров Intel и Apple Silicon.
  Приложение не подписано сертификатом Apple Developer ID и не проходит нотариализацию, поэтому при первом запуске
  необходимо воспользоваться пунктом «Открыть» в контекстном меню Finder либо снять атрибут карантина командой:

```sh
xattr -cr "/Applications/Amnezia-WG Admin.app"
```

- **Linux** — исполняемый файл `amnezia-admin_<версия>`. Для запуска необходимы системные библиотеки `libgtk-3` и
  `libwebkit2gtk-4.1`.

Desktop-приложение не публикует сетевой интерфейс и не требует аутентификации пользователей.

### Standalone-сервер

Standalone-режим предназначен для эксплуатации приложения в виде веб-сервиса. Пользовательский интерфейс становится
доступен через браузер, а административное приложение может выполняться на сервере без графической среды.

После запуска веб-сервер начинает принимать HTTP-запросы на адресе `:8080`, если не указано иное.

Запуск из бинарного файла:

```sh
./awg-admin
```

Запуск в контейнере Docker:

```sh
docker run -d --name awg-admin \
  -p 8080:8080 \
  -v awg-admin-data:/data \
  ghcr.io/<owner>/<repo>:latest
```

При первом запуске используются учётные данные:

- логин: `admin`
- пароль: `admin`

После входа рекомендуется заменить пароль в разделе **Settings** или через API:

```http
PUT /auth/change-credentials
```

Механизм аутентификации используется только в standalone-режиме. Desktop-приложение выполняется локально и не
предоставляет удалённый доступ по сети.

### Конфигурация standalone-сервера

Поведение веб-сервера настраивается через переменные окружения.

| Переменная                                 | Назначение                                                                                                                                                                                           |
|--------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `AWG_ADMIN_ADDR`                           | Адрес прослушивания HTTP-сервера. Значение по умолчанию — `:8080`.                                                                                                                                   |
| `AWG_ADMIN_TLS_CERT` / `AWG_ADMIN_TLS_KEY` | Использование заранее подготовленного TLS-сертификата и закрытого ключа.                                                                                                                             |
| `AWG_ADMIN_AUTOCERT_DOMAINS`               | Список доменов, для которых сертификат должен быть автоматически получен через Let's Encrypt. Не может использоваться одновременно со статическим сертификатом. Требуется доступность HTTP-порта 80. |
| `AWG_ADMIN_AUTOCERT_CACHE_DIR`             | Каталог хранения автоматически выпущенных сертификатов. По умолчанию используется `$HOME/.awg-admin-autocert`.                                                                                       |

### Каталоги данных

Standalone-приложение использует каталог `$HOME` для хранения рабочего состояния. По умолчанию создаются следующие файлы
и каталоги:

| Путь                        | Назначение                                        |
|-----------------------------|---------------------------------------------------|
| `$HOME/.awg-admin`          | база данных BoltDB с конфигурацией инфраструктуры |
| `$HOME/.awg-admin-autocert` | кэш автоматически выпущенных TLS-сертификатов     |
| `$HOME/.awg-admin-cache`    | локальный кэш загруженных бинарных файлов агента  |

В официальном Docker-образе в качестве домашнего каталога используется `/data`. Этот каталог предполагается подключать
как постоянный volume.

### Агент

Агент представляет собой отдельный исполняемый файл, предназначенный для Linux. Он управляет реальными интерфейсами
WireGuard и поэтому должен выполняться непосредственно на управляемом сервере. Обычно агент устанавливается
автоматически
административным приложением через SSH (см. [Развёртывание агента](#развёртывание-агента)). При необходимости его можно
запустить вручную:

```sh
AWG_AGENT_ADDR=127.0.0.1:8080 AWG_AGENT_DB=/var/lib/awg-agent ./awg-agent
```

После запуска агент начинает обслуживать HTTP API и ожидает подключения административного приложения. По умолчанию API
публикуется только на loopback-интерфейсе (`127.0.0.1`) и рассчитано на использование через SSH-туннель. При
необходимости агент может принимать соединения напрямую по mTLS.

### Конфигурация агента

| Переменная                                                             | Назначение                                                                                           |
|------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------|
| `AWG_AGENT_ADDR`                                                       | Адрес прослушивания HTTP-сервера. По умолчанию `127.0.0.1:8080`.                                     |
| `AWG_AGENT_DB`                                                         | Каталог хранения локальной конфигурации интерфейсов.                                                 |
| `AWG_AGENT_TLS_CERT` / `AWG_AGENT_TLS_KEY` / `AWG_AGENT_TLS_CLIENT_CA` | При указании всех трёх переменных агент использует mTLS вместо обычного HTTP.                        |
| `AWG_AGENT_METRICS_INTERVAL`                                           | Интервал сбора статистики процессора, памяти, сети и WireGuard-пиров. Значение по умолчанию — `45s`. |
| `AWG_AGENT_LOG_LEVEL`                                                  | Уровень логирования. По умолчанию — `info`.                                                          |

Агент поддерживает только Linux и распространяется в сборках для архитектур `amd64` и `arm64`.

## Как всё устроено

Amnezia-WG Admin оперирует четырьмя типами сущностей, вложенных друг в друга:

- **Сервер** — удалённая машина, доступная по SSH. Amnezia-WG Admin хранит параметры её подключения и, после развёртывания
  агента, взаимодействует с агентом, выполняющимся на ней.
- **Интерфейс** — сетевой интерфейс WireGuard/AmneziaWG на сервере (`wg0`, `awg0` и т. д.). У сервера их может быть
  несколько.
- **Пользователь** — человек или владелец устройства. Пользователи существуют независимо от серверов.
- **Пир** — отдельный VPN-клиент, принадлежащий пользователю и привязанный к одному интерфейсу на одном сервере. Пиры
  выдаются клиентам в виде конфигурационных файлов или QR-кодов.

Административное приложение хранит все эти данные — вместе с SSH-учётными данными, ключами и привязками — в локальной
встроенной базе (BoltDB) и рассматривает её как единственный источник истины. Агент лишь применяет переданную ему
конфигурацию интерфейса и не хранит собственного желаемого состояния. Любое изменение сначала сохраняется локально, а
затем передаётся соответствующему агенту по принципу best-effort: временно недоступный агент не блокирует правку —
интерфейс помечается как *не синхронизированный* до следующей успешной передачи или ручной команды **Sync**.

## Серверы

### Добавление сервера

Сервер добавляется на странице **Серверы**. Как минимум требуются SSH-хост и адрес агента. Необязательные метаданные —
**имя**, **описание**, **локация** и **теги** — используются только для организации и поиска.

Создание сервера не выполняет к нему подключение: форма лишь записывает параметры подключения, поэтому серверы можно
добавлять до того, как они станут доступны. Ошибки соединения (неверный ключ, неправильная passphrase, недоступный хост)
проявляются позже — при выполнении операций **Deploy**, **Sync** или **Update**, а не при создании.

### SSH-аутентификация

Amnezia-WG Admin подключается к серверу по SSH для развёртывания агента и, по умолчанию, для проброса туннеля к его API.
Поддерживаются следующие типы учётных данных:

- **Файл приватного ключа** — путь к ключу на машине, где выполняется Amnezia-WG Admin.
- **Загруженный ключ** — содержимое ключа, переданное напрямую и сохранённое в базе данных. Это переносимый вариант: он
  работает одинаково в desktop-приложении и во вкладке браузера и не зависит от пути к файлу, существующего лишь на
  одной
  машине. Если заданы и путь, и содержимое, приоритет имеет содержимое.
- **Пароль** — обычная парольная SSH-аутентификация.

**Ключи, защищённые passphrase.** Если ключ требует passphrase, первая операция, подключающаяся к серверу, запрашивает
её. Passphrase кэшируется в памяти на время жизни процесса и не записывается на диск, поэтому запрашивается только один
раз за сеанс. Параметр *Использовать для всех подключений* переиспользует её как запасной вариант для других серверов,
чьи ключи защищены той же passphrase.

Ключи хоста не проверяются (заранее заданный `known_hosts` отсутствует), что соответствует поведению лёгких инструментов
управления серверами при первом подключении.

### Доступ к агенту: SSH-туннель или mTLS

Когда агент запущен, административное приложение взаимодействует с его HTTP API одним из двух способов:

- **SSH-туннель (по умолчанию).** Агент слушает на loopback (`127.0.0.1:8080`) и не выставляется в сеть; приложение
  обращается к нему через долгоживущее SSH-соединение, открытое на весь сеанс. Дополнительная настройка не требуется —
  достаточно указать SSH-параметры сервера.
- **Прямой mTLS.** Предназначен для агента, к которому предпочтительно обращаться напрямую по публичному IP без SSH.
  Действие **Generate TLS** на странице сервера выпускает набор сертификатов CA, сервера и клиента; агент затем
  запускается с этими сертификатами и проверяет клиентский сертификат приложения при каждом запросе. Этот способ
  уместен,
  когда удобный SSH-путь отсутствует или когда приложение и агент должны быть полностью развязаны с SSH.

Соединение отказоустойчиво: если туннель обрывается незаметно (перезагрузка хоста или сетевой сбой), приложение
обнаруживает это и переподключается автоматически при следующем вызове.

### Окно агента

Все операции, выполняемые над самим агентом (в отличие от обслуживаемых им интерфейсов и пиров), собраны в едином окне
**агента**, а не разбросаны по странице. Оно открывается кнопкой **«гаечный ключ»** на странице сервера или кнопкой
**«шестерёнка»** в строке сервера на дашборде. В верхней части окна отображаются сообщённые агентом **версия** и
возможности (бэкенд, Docker, типы интерфейсов); ниже расположены все действия с агентом:
[развёртывание](#развёртывание-агента), [синхронизация](#синхронизация-sync), [сверка](#сверка-агент--база-данных),
[переключатель мониторинга](#переключатель-мониторинга) и [профилирование](#профилирование). Каждое описано в разделах
ниже.

### Развёртывание агента

Действие **Deploy agent** в окне агента устанавливает и запускает агента по SSH за один шаг: кросс-скомпилированный
Linux-бинарник загружается и запускается автоматически. После развёртывания приложение заново передаёт все интерфейсы
этого сервера, чтобы хост поднялся в желаемом состоянии. Область статуса отображает прогресс и любые ошибки.

В desktop-приложении локальный бинарник агента также можно выбрать с диска через нативный диалог выбора файла, не
полагаясь на скачанный.

#### Требования на сервере

Два выбора независимы: **бэкенд** агента (ядро или userspace — способ создания линка интерфейса) и **метод
развёртывания** (systemd или Docker, определяемый видом [источника агента](#источники-агента-пресеты-деплоя) — источник
**URL** или **локальный файл** устанавливает бинарник как сервис **systemd**, а источник **образ** запускает
**контейнер**). Требования к серверу зависят от используемого сочетания:

- **`awg-agent` — бэкенд ядра (systemd).** Работает через модуль ядра AmneziaWG напрямую и потому требует его установки
  на сервере:
    - **модуль ядра AmneziaWG** для интерфейсов AmneziaWG — установите `amneziawg-dkms` согласно
      [README amneziawg-linux-kernel-module](https://github.com/amnezia-vpn/amneziawg-linux-kernel-module/blob/master/README.md)
      и загрузите его (`modprobe amneziawg`). Развёртывание проверяет это заранее и завершается с понятным сообщением,
      если
      модуль отсутствует;
    - **WireGuard**, если также создаются обычные (не-Amnezia) интерфейсы — см.
      [wireguard.com/install](https://www.wireguard.com/install/).
- **`awg-agent-userspace` — userspace-бэкенд (systemd или Docker).** Встроенная реализация userspace-WireGuard
  (библиотека amneziawg-go вкомпилирована), не требующая модуля ядра; предназначена для хостов, где модуль установить
  нельзя или нежелательно. Разворачивается одним из двух способов:
    - **как бинарник через systemd** — с помощью источника агента вида **URL** или **локальный файл**, указывающего на
      бинарник `awg-agent-userspace`, с включённым параметром **Userspace-агент (без модуля ядра)**. Устанавливается как
      сервис systemd точно так же, как агент ядра, и требует только `/dev/net/tun` (стандартно для Linux); Docker не
      задействуется. Этот параметр указывает развёртыванию пропустить проверку модуля ядра, которая иначе применяется к
      агенту ядра.
    - **как контейнер** — с помощью источника вида **образ**. Развёртывание запускает контейнер с `/dev/net/tun` и
      `NET_ADMIN` и заранее проверяет `docker info`. Требуется только **Docker** — см.
      [docs.docker.com/engine/install](https://docs.docker.com/engine/install/).

Обе сборки предоставляют одинаковый API; различается лишь способ создания линка интерфейса. Версии должны совпадать:
модуль ядра AmneziaWG 1.0 отвергает параметры обфускации в формате 2.0 (диапазоны `H1–H4`, `I1–I5`). В этом случае
обновите модуль до версии 2.0 либо деактивируйте интерфейс (см. [Интерфейсы](#интерфейсы)), чтобы сохранить его
конфигурацию без применения.

После запуска агент определяет, что реально поддерживает его хост — Docker, модуль ядра AmneziaWG, какие типы
интерфейсов
он может создавать и запущен ли он в контейнере — и сообщает об этом; колонка **типа агента** на
[Дашборде](#дашборд-и-метрики) это отображает.

### Источники агента (пресеты деплоя)

**Источник агента** — переиспользуемый именованный пресет, описывающий, откуда получить бинарник агента (например, URL
релиза), с необязательным локальным кэшированием, чтобы повторные развёртывания не скачивали его заново. Источники
агента
управляются из выпадающего списка в разделе развёртывания окна агента: пункт **«Добавить новый…»** открывает небольшую
форму создания, а у каждой сохранённой записи есть действие **изменения** (карандаш) для просмотра и правки URL, пути,
образа и параметров, а также действия обновления кэша и удаления. Это удобно при развёртывании на множество серверов или
при фиксации конкретной версии агента.

Источник бывает одного из трёх видов — **URL**, **локальный путь к файлу** или **Docker-образ** (последний разворачивает
userspace-агента как контейнер). Если агент уже сообщил, что на сервере нет работающего Docker, источники-образы для
него
не предлагаются: они скрываются из списка, а вариант **Docker-образ** убирается из формы создания, поскольку
контейнерное
развёртывание там невозможно. До первого развёртывания возможности сервера неизвестны, поэтому предлагаются все
варианты,
а подстраховкой служит собственная проверка Docker при развёртывании.

### Синхронизация (Sync)

**Sync** заново передаёт агенту все интерфейсы, записанные для сервера, перезаписывая текущее состояние агента желаемым
состоянием из базы данных. Она выполняется автоматически после развёртывания и после переподключения туннеля; ручное
действие предусмотрено для принудительного возврата агента в соответствие — например, после его переустановки или при
подозрении на расхождение.

### Сверка (агент ↔ база данных)

**Сверка** (Reconcile) сравнивает то, что реально настроено у агента, с записями базы данных по имени интерфейса и
показывает расхождения, чтобы каждое можно было разрешить вручную. Возможны два вида расхождений:

- **Присутствует у агента, но отсутствует в базе данных** (например, после восстановления базы из старой резервной
  копии): интерфейс можно **импортировать** в базу данных или **удалить с агента**.
- **Присутствует в базе данных, но отсутствует у агента** (например, после того как агент потерял своё хранилище или был
  переустановлен): интерфейс можно **заново передать** агенту или **удалить из базы данных**.

Когда обе стороны согласованы, сверка сообщает, что действий не требуется. Следует учитывать, что импорт восстанавливает
только сам интерфейс (адрес, ключи, параметры AmneziaWG и пиры, о которых сообщает агент); связь пира с пользователем,
которому он принадлежит, существует только в административной базе данных и не может быть восстановлена из агента.

### Переключатель мониторинга

У каждого сервера есть переключатель мониторинга, включающий или отключающий сбор метрик агентом в реальном времени.
Желаемое состояние сохраняется и заново применяется при передеплое, поэтому сервер, у которого мониторинг отключён,
остаётся таковым и после перезапуска агента.

### Профилирование

Для диагностики некорректно работающего агента **профилирование** открывает доступ к профилированию Go-рантайма агента
(`net/http/pprof`). По умолчанию оно отключено, поскольку сэмплирование создаёт накладные расходы, а эндпоинты
раскрывают
внутреннее состояние, и включается по каждому серверу из окна агента; как и мониторинг, желаемое состояние заново
применяется при передеплое. Пока профилирование включено, рядом с именем сервера (на дашборде и на странице сервера)
отображается небольшой значок активности в качестве напоминания.

При включённом профилировании действие **Скачать дамп** получает профиль от агента для анализа с помощью `go tool pprof`
или `go tool trace`. Доступные виды: `goroutine` (зависшие или утекающие горутины), `heap` (память), CPU-профиль
`profile` и трассировка выполнения `trace` (оба сэмплируются в течение заданного числа секунд), а также стандартные
`allocs`, `block`, `mutex` и `threadcreate`. Desktop-приложение сохраняет дамп через нативный диалог; веб-приложение
скачивает его в браузере. По завершении диагностики профилирование следует отключить.

## Интерфейсы

Страница сервера позволяет создавать, редактировать и удалять интерфейсы WireGuard/AmneziaWG. Форма состоит из двух
вкладок.

Вкладка **Основные**:

- **Имя** — имя интерфейса (`wg0`, `awg0` и т. д.). Обязательно.
- **Адрес** — собственный адрес интерфейса в формате CIDR (например, `10.0.0.1/24`). Необязательно: если оставить поле
  пустым, Amnezia-WG Admin назначает первый хост следующей свободной `/24` из пула `172.23.0.0/16`, выбранной так, чтобы
  она
  не пересекалась ни с одним существующим интерфейсом. При редактировании пустое поле сохраняет текущий адрес, поэтому
  интерфейс не переносится в новую подсеть, а его пиры не осиротеют.
- **Порт прослушивания** — UDP-порт, который слушает интерфейс (по умолчанию `51820`).
- **Приватный ключ** — оставьте пустым для автоматической генерации.
- **Активировать интерфейс** — флажок, установленный по умолчанию. Пока он установлен, агент поднимает интерфейс и
  поднимает его заново при каждом перезапуске агента. Снятие флажка сохраняет конфигурацию, но предписывает агенту
  опустить интерфейс; в списке у интерфейса появляется бейдж **«Отключён»**. Это позволяет «припарковать» интерфейс, не
  удаляя его.

Вкладка **Amnezia** содержит значения `Jc/Jmin/Jmax`, `S1–S4`, `H1–H4` и `I1–I5` для junk-пакетов и обфускации
заголовков, затрудняющие фингерпринтинг трафика AmneziaWG:

- **Интерфейс Amnezia** — флажок в верхней части вкладки, установленный по умолчанию. Пока он установлен, все параметры
  обфускации ниже предзаполнены свежесгенерированными значениями (тем же набором, который приложение применило бы
  самостоятельно), и каждое поле остаётся доступным для ручной правки.
- Снятие флажка создаёт вместо этого обычный интерфейс **WireGuard**: параметры обфускации при этом игнорируются, и
  интерфейс работает как стандартный WireGuard.

При редактировании вкладка отражает текущую конфигурацию интерфейса — флажок установлен (и показаны сохранённые
значения) для интерфейса AmneziaWG и снят для обычного WireGuard.

Новый или изменённый интерфейс проверяется относительно существующих интерфейсов сервера: его **имя**, **порт
прослушивания** и **подсеть** (производная от адреса) должны быть свободны. Дублирующее имя или порт, либо подсеть,
пересекающаяся с другим интерфейсом, отклоняются с указанием причины. При редактировании неизменённое поле повторно не
проверяется.

Интерфейсы, входящие в [туннель](#многоузловые-туннели-exit-ноды), заблокированы для редактирования и удаления в
пользовательском интерфейсе; сначала необходимо удалить туннель.

Удаление интерфейса также удаляет все его пиры, поскольку пир не может существовать без своего интерфейса. Если у
интерфейса есть пиры, пользовательский интерфейс требует подтверждения дважды, и первый диалог сообщает, сколько пиров
будет удалено. Удалённые пиры также перестают отображаться в [метриках пиров](#дашборд-и-метрики).

Каждый интерфейс отслеживает статус **синхронизации**: успешна ли была последняя передача агенту, ошибку (если нет) и
время последней успешной синхронизации.

При запуске агент поднимает только активные интерфейсы, а при остановке опускает все активные, поэтому остановленный
агент не оставляет работающих туннелей. Деактивированный интерфейс не поднимается, и его конфигурация сохраняется, но не
применяется — что также делает деактивацию способом «припарковать» интерфейс, который агент иначе не смог бы
сконфигурировать.

Если интерфейсу не задан адрес IPv6, он поднимается с **отключённым IPv6** (туннели здесь работают только по IPv4),
поэтому ему не назначается автоматический link-local IPv6-адрес, и он не может нести или «утекать» трафик IPv6.
Интерфейс, которому намеренно задан IPv6-адрес, остаётся без изменений. Это делается по принципу best-effort: там, где
агент не может применить настройку — в частности, в Docker-контейнере userspace-агента с read-only `/proc/sys`, —
интерфейс всё равно поднимается, а в лог записывается предупреждение.

### Hook-команды

Каждый интерфейс поддерживает hook-команды **PreUp**, **PostUp**, **PreDown** и **PostDown**, выполняемые на агенте при
поднятии или опускании интерфейса, — аналог hook-команд wg-quick. Поскольку агент не запускает wg-quick, всё, что
относится к произвольной маршрутизации или правилам `iptables`, выражается через hook-команды. В команде `%i`
разворачивается в имя интерфейса. Именно эти hook-команды функция туннелей использует внутренне для настройки политики
маршрутизации и NAT.

## Пользователи и пиры

**Пользователи** (страница *Users*) — это люди, которым принадлежат пиры. У пользователя есть **имя**, необязательное
**описание**, и его можно **отключить**. Пользователи не зависят от серверов: у одного пользователя могут быть пиры на
нескольких серверах.

**Пир** — отдельный VPN-клиент, принадлежащий пользователю. При добавлении пира:

- **Имя** — метка пира (например, название устройства).
- **Сервер** и **интерфейс** — место привязки пира.
- **Allowed IPs** — адреса пира; оставьте пустым для автоматического назначения свободного хост-адреса в подсети
  интерфейса. Указанный хост-адрес (`/32`) должен находиться внутри подсети интерфейса и не должен быть уже занят другим
  пиром или самим интерфейсом; адреса вне подсети и дубликаты отклоняются. Более широкие маршрутные CIDR — LAN за
  site-to-site пиром, например `192.168.1.0/24`, — допускаются вне подсети.
- **Endpoint** — необязательно, для пиров, к которым следует подключаться по фиксированному адресу.
- **DNS** — необязательный клиентский DNS для генерируемой конфигурации этого пира (один или несколько серверов через
  запятую, например `1.1.1.1, 8.8.8.8`). Становится строкой `DNS = …` в клиентской конфигурации; если поле пустое,
  используется DNS интерфейса. Это клиентская настройка, агенту она не передаётся.
- **Приватный ключ** — необязательно; если поле пустое, для пира генерируется новый ключ. Заполняется только для импорта
  существующего ключа.
- **Pre-shared key** — необязательный дополнительный симметричный ключ для усиления защиты. Параметр *Сгенерировать
  preshared key* включён по умолчанию и создаёт случайный ключ. Его отключение открывает поле для ввода собственного
  ключа; это поле можно оставить пустым, чтобы создать пир без pre-shared key.

Для каждого пира приложение предоставляет:

- **Конфиг** — готовый текст клиентской конфигурации `wg-quick`/AmneziaWG с ключами пира, endpoint сервера и параметрами
  обфускации AmneziaWG.
- **QR-код** — та же конфигурация в виде сканируемого QR-кода для мобильных приложений, с действием сохранения в файл
  PNG
  (нативный диалог сохранения в desktop-приложении).

Пир можно переместить на другой интерфейс действием перемещения (⇄) у пира с последующим выбором цели на любом сервере:
его ключи, имя, DNS и pre-shared key сохраняются. Адрес сохраняется, если он по-прежнему подходит под подсеть целевого
интерфейса и там свободен — например, при перемещении между двумя членами туннеля, у которых общая подсеть, — иначе на
цели назначается свободный адрес. Оба интерфейса передаются заново.

Пир можно деактивировать переключателем (⏻) у пира, не удаляя его. Деактивированный пир сохраняет свою конфигурацию —
ключи, адрес и pre-shared key, — но убирается с активного интерфейса на агенте, поэтому этот клиент больше не может
подключиться, пока не будет активирован снова. Это аналог переключателя *Активировать* на уровне интерфейса, применённый
к отдельному клиенту, и удобен для временного отзыва доступа без потери пира или его адреса. Конфигурация и QR-код
остаются доступными, но деактивированный пир отсутствует на устройстве и потому не показывает трафика в метриках.

Удаление пира отзывает его на агенте (убирая из интерфейса) и удаляет из метрик пиров, поэтому он больше не отображается
в истории метрик.

## Многоузловые туннели (exit-ноды)

**Туннель** связывает два сервера так, что клиенты, подключённые к первому, выходят в интернет через второй. Типичный
сценарий: **relay** (сервер 1), к которому подключаются клиенты, и **exit-нода** (сервер 2), чей публичный IP видит
интернет.

Туннель создаётся на странице **Tunnels** с помощью мастера:

1. **Вход (Entry)** — выберите сервер и интерфейс, к которому клиенты уже подключаются (relay). У него должны быть порт
   прослушивания и CIDR-адрес.
2. **Выход (Exit)** — выберите второй сервер и интерфейс, который станет выделенной exit-нодой. Его конфигурация
   заменяется под эту роль.
3. **Обзор** — для подтверждения показывается общая подсеть туннеля. Это всегда собственная подсеть входного интерфейса
   (на неё помещается выход), а не задаваемое вручную значение.

Amnezia-WG Admin перенастраивает оба интерфейса и передаёт их: relay получает правила policy-routing и gateway-пира,
указывающего на выход; выход получает NAT (masquerade) и сам подключается к relay. Клиентам не требуется менять
конфигурацию — их существующий туннель до relay теперь выходит через второй сервер.

Поскольку выход находится в подсети входа, оба члена используют единый пул адресов: пир, добавленный на любом из концов,
получает адрес, уникальный для обоих. Клиентские пиры можно добавлять как на интерфейс выхода, так и на relay.

Удаление туннеля возвращает оба интерфейса в пустое состояние (убирая добавленных пиров, hook-команды и маршрутизацию),
а
не удаляет их, поэтому интерфейсы остаются доступными для повторного использования. Пока туннель существует, два его
интерфейса нельзя редактировать или удалять по отдельности, как и серверы, на которых они находятся; сначала необходимо
удалить туннель.

## Дашборд и метрики

**Дашборд** показывает итоговые значения (серверы, пиры, пользователи, туннели) и таблицу по серверам с бейджем *
*статуса
агента**, колонкой **типа агента**, счётчиком **пиров**, показателем **load average (1/5/15)** и **% RAM**. Выбор
сервера
открывает **окно метрик** с двумя вкладками:

- **System** — CPU, RAM, нагрузка и сеть хоста во времени.
- **Peers** — таблица активности по каждому пиру с компактным спарклайном его трафика, с подписью
  `<пользователь>/<пир>`.

Бейдж **статуса агента** — трёхцветная проверка работоспособности, которая опрашивает агента и объединяет результат с
состоянием соединения с ним:

- **Зелёный** — агент доступен и отвечает (для сервера через SSH-туннель это также подтверждает, что туннель поднят).
- **Красный** — соединение не работает: SSH-туннель не удалось поднять либо mTLS-агент, доступ к которому осуществляется
  напрямую, недоступен.
- **Жёлтый** — промежуточное состояние, например, когда SSH-туннель поднят, но агент за ним не отвечает.

До завершения первой проверки отображается серое состояние *неизвестно*. Любое не-зелёное состояние записывается в лог
на
уровне error, поэтому проблемы видны в логах агента и админки.

Колонка **типа агента** отражает то, что агент определил о своём хосте при запуске: его сборку-бэкенд (`kernel` или
`userspace`), индикатор **docker**, когда сам агент запущен в контейнере, и типы интерфейсов, которые он может там
создавать (`awg` для AmneziaWG, `wg` для обычного WireGuard). Наведение на колонку показывает полную информацию —
доступен ли Docker на хосте, присутствует ли модуль ядра AmneziaWG и версию агента. Это позволяет с одного взгляда
понять,
может ли сервер обфусцировать трафик и как запущен его агент. Как и метрики, эти данные предоставляются по принципу
best-effort: пока агент ненадолго недоступен, показываются последние известные значения, а до первого успешного ответа —
`—`.

Метрики формируются агентом, который сэмплирует хост и rx/tx/handshake каждого пира с заданным интервалом (по умолчанию
45с, настраивается через `AWG_AGENT_METRICS_INTERVAL`) в кольцевые буферы в памяти, сохраняя до 48 часов истории.
История
периодически сбрасывается на диск агента, поэтому графики переживают перезапуск агента. Метрики предоставляются по
принципу best-effort: сервер, чей агент ненадолго недоступен, показывает последние известные значения, а доступность
сигнализируется отдельно бейджем статуса агента.

Агент также может отдавать метрики в текстовом формате **Prometheus** (эндпоинт `/metrics?fmt=prom`) для сбора во
внешнюю
систему мониторинга.

## Настройки

Страница **Settings** подстраивается под режим развёртывания:

- **Язык** — English или Русский. Доступно в обоих режимах.
- **Резервная копия** — описано ниже. Доступно в обоих режимах.
- **Логи** и **режим отладки** — только desktop.
- **Basic Auth** и **смена учётных данных** — только standalone.
- **Версия** — версия сборки приложения.

### Резервная копия

**Сохранить копию** скачивает полный снимок административной базы данных — серверы, пользователи, пиры, интерфейсы и
учётные данные — в виде файла JSON. В desktop-приложении открывается нативный диалог сохранения; в standalone-сервере
файл скачивается через браузер. Файл использует тот же переносимый формат, что и утилита `awg-migrate`, и может быть
восстановлен командой `awg-migrate import` (восстановления внутри приложения нет, поскольку это заменило бы базу данных
под работающим процессом — см. [Бэкап, восстановление и перенос](#бэкап-восстановление-и-перенос)).

> Резервная копия содержит секреты — приватные SSH-ключи, mTLS-ключи агента, pre-shared ключи пиров и хеш пароля
> администратора. Храните её в надёжном месте.

### Логи и режим отладки (desktop)

Desktop-приложение захватывает собственный вывод логов в память. **Просмотр логов** показывает последние строки лога и
позволяет сохранить их в файл JSON, что удобно для баг-репортов. Флажок **Режим отладки** включает логирование уровня
debug в реальном времени; по умолчанию он выключен, чтобы не переполнять лог. Предполагаемый порядок действий: включить
его, воспроизвести проблему и затем обновить, чтобы увидеть подробные строки. В standalone-режиме буфер логов
отсутствует, поэтому этот раздел доступен только в desktop.

### Вход, учётные данные и Basic Auth (standalone)

Standalone веб-сервер — единственный режим с аутентификацией; desktop-приложение выполняется локально и не имеет сетевой
поверхности, которую нужно защищать.

- **Вход** — единственная учётная запись администратора, при первом запуске — **admin / admin**. Её следует сменить
  немедленно.
- **Смена учётных данных** — задаёт новые логин и пароль.
- **HTTP Basic Auth** — необязательный дополнительный барьер: при включении каждый запрос (включая саму страницу входа)
  дополнительно требует HTTP Basic-аутентификацию с той же учётной записью администратора. Это удобно при прямом
  выставлении сервера без reverse-proxy. По умолчанию выключено.

## Бэкап, восстановление и перенос

Всё административное состояние находится в одном файле BoltDB (`~/.awg-admin`). С ним работают два инструмента,
использующие один и тот же переносимый формат JSON-дампа:

- **Резервная копия в приложении** (Settings → Резервная копия) — экспорт текущего состояния в файл JSON за один шаг.
- **`awg-migrate`** — консольная утилита экспорта и импорта, используемая для переноса данных между машинами или между
  desktop- и standalone-режимами (оба читают один формат файла):

  ```sh
  awg-migrate export -db ~/.awg-admin -out dump.json   # на исходной установке
  awg-migrate import -db ~/.awg-admin -in dump.json    # в целевую установку
  ```

Экспортируемый документ содержит полную административную конфигурацию — управляемые серверы, SSH-учётные данные,
пользователей, VPN-пиров и их взаимосвязи. Поскольку оба режима развёртывания используют одинаковую модель данных, один
и
тот же экспорт может использоваться для перехода между desktop- и standalone-установками или для переноса приложения на
другую машину.

Файл, сохранённый через **Резервную копию** в приложении, является корректным дампом `awg-migrate`. Чтобы восстановить
резервную копию, импортируйте её через `awg-migrate` в новую или целевую базу данных при остановленном приложении, а
затем запустите приложение.

## Где хранятся данные и секреты

- **Административная база данных** — `~/.awg-admin` (BoltDB). Содержит всё: серверы, SSH-учётные данные и ключи,
  пользователей, пиры, конфигурации интерфейсов и учётную запись администратора для standalone-режима. Именно её
  необходимо резервировать.
- **Кэш autocert** (HTTPS в standalone) — `~/.awg-admin-autocert`.
- **Кэш бинарников агента** (пресеты деплоя) — `~/.awg-admin-cache`.
- **Агент, на каждом сервере** — хранит только переданные ему конфигурации интерфейсов (`AWG_AGENT_DB`, по умолчанию
  `/var/lib/awg-agent`) вместе с чекпоинтом истории метрик. Никаких административных секретов, кроме ключей интерфейсов
  и
  пиров, необходимых для работы WireGuard, он не хранит.

В Docker-образе standalone-режима `$HOME` соответствует `/data`, поэтому монтирование этого volume сохраняет всё
перечисленное.

Hook-команды интерфейса выполняются на хосте агента с его привилегиями (обычно root, так как он управляет сетевыми
интерфейсами) — это сделано намеренно, ровно как hook-команды wg-quick. Они исходят от администратора и передаются по
аутентифицированному каналу.
